Linux-Coding-Style

发表于 2018-02-06 | 分类于 学海无涯

缩进

  • 文档推荐的是缩进是8个字符，理由就是8个宽度容易重视函数嵌套太深的问题，如果超过3级缩进，则可能有代码问题了。 但是实际使用的时候还是觉得使用4个字符比较合适。

    switch 语句中消除缩进的方式可以这样：
switch  (suffix) {
case 'g':
case 'G':
    mem <<= 30;
    break;    
default:
    break;    
}

  • 不要把多个语句放在一行里，除非你有什么东西要隐藏，也不要在一行里放多个赋值语句

  • 除了注释、文档和Kconfig之外，不要使用空格来缩进
  • 不要在行尾留空格，空白

保持一行的长度

  • 每一行的长度的限制是80列
  • 保持较长的显示，除非是可以大幅度的提高可读性，并且不会隐藏信息

大括号和空格

  • 大括号的放置，起始括号在行末，结束括号在行首。这适用所有的非函数语句块 (if、 switch、 for、 while、 do)

    if （x is true） {
    we do y
}

    不过例外的函数括号，一般放置在下一行的开头。大家可能会抱怨这个不一致性。但是函数确实是特殊的

    int function(int x)
{
    body of function
}

do {
    body of do-loop
} while (condition);

  • 空格的使用，取决于是函数还是关键字。 （大多数）关键字后要加一个空格，例外的是 sizeof, typeof, alignof, __attribute__ 因为他们看起来更像函数。列如：

    s = sizeof(struct file);

    所以在这些关键字后面放一个空格：if, switch, case, for, do, while。但不要在小括号里的表达式两侧加空格，比如这个反列: s = sizeof( struct file );

    “*” 的使用，当声明指针类型或者返回指针类型的函数时，“*” 是使之靠近变量名或者函数名，而不是靠近类型名。比如:

    char *linux_banner;
unsigned long long memparse(char *ptr, char **retptr);
char *match_strdup(substring_t *s);

  • 大多数的二元，三元操作符两侧使用一个空格：

    = + - < > * / % | & ^ <= >= == != ? :

    但是一元操作符后不要加空格：

    & * + - ~ ! sizeof typeof alignof `__attribute__` defined ++ --

当然结构体成员操作符前后也不需要空格： `“ .”和“ ->”`

命名

  • C 是一个简谱的语言
  • C 不推荐使用类似于 ThisVariableIsATemporaryCounter 这样华丽的名字，却更倾向于 tmp； C 不提倡使用大小写混用的方式来命名，但对于全局变量还是需要一个具体的描述性名字。
  • 全局变量，只有正常需要他们的时候使用它
  • 全局变量，需要一个具体的名字，比如计算用户活动量的数量函数，你应该叫它 count_active_users() 类似的名字，而不是 cntuser（）。
  • 不提倡在函数名中包含函数类型（所谓的匈牙利命名法）
  • 本地变量名应该简短，循环计数变量，称之为 i 也是不错的。

Typedef

  • 不推荐对结构体和指针使用typedef， 类似 vps_t a;而推荐使用 struct virtual_container *a,这样比较明确变量类型和作用。

  • typedef 一般在下列情况下使用：

    1. 完全不透明的对象，主动使用 typedef 来隐藏 这个对象。只能用合适的访问函数来访问。（不透明和访问函数本身是不好的。）

    2. 清除的整数类型，比如，这层抽象可以帮助消除 int 还是 long 的混淆。u8/u16/u32 是完全没有问题的 typedef。 一般情况下如果变量是 unsigned long ，那么没有必要

      typedef unsigned long myflags_t;

      不过有一个明确的原因，比如某些情况下是 unsigned int，其他情况下是 unsigned long，此时就需要用typedef

    3. 标准C99类型，有些人排斥 uint32_t,而倾向于 u8/u16/u32/u64，因为需要尊重那些已经做出选择的代码。
    4. 可以在用户空间安全使用的类型

函数

  • 函数应该简短漂亮，只完成一件事。

  • 一个函数的最大长度，应该和该函数的复杂度，缩进级数成反比。（理论上很简单的只有一个很长的case语句，而且需要在case里做很多小的事情，这个函数尽管很长，但也是可以的）

    如果一个复制的函数，甚至难以理解函数的目的，应该使用辅助函数，并取一个可以描述的名字（如果性能很重要，可以让编译器内联他们，这样的效果往往会比你写一个复杂的效果要好。）

  • 函数本地变量的数量，不能超过5-10个，否则需要拆分成更小的函数

  • 人的大脑一般可以轻松的同时跟踪7个不同的事物，更多就会变得糊涂；在聪明的人也可能会记不清2个星期前做过的事情。
  • 在源文件里，使用空行隔开不同的函数。
  • 函数需要被导出，它的EXPORT*宏应该紧贴在它的结束大括号之下。

goto的使用

  • 必要的时候使用 goto 有好处 理由是：
    • 无条件语句容易理解和跟踪
    • 嵌套程度减小
    • 可以避免由于修改时忘记更新某个单独的退出点而导致的错误
    • 减轻了编译器的工作，无需删除冗余代码;)

注释

  • 注释是好的，不过有过度注释的危险
  • 永远不要在注释里解释你的代码是如何运作的：更好的做法是让别人一看你的代码就可以明白，解释写的很差的代码是浪费时间。
  • 注释放在函数的头部，不要放在函数内部。
  • 注释告诉别人你做了什么，还可以加上做这些事情的原因。

  • 注释风格 提倡C89 /*...*/，不提倡 C99 //... 

    /*
* This is the preferred style for multi-line
* comments in the Linux kernel source code.
* Please use it consistently.
*
* Description: A column of asterisks on the left side,
* with beginning and ending almost-blank lines.
*/
  • 注释数据也是很重要的,每一行声明一个数据，这样可以为每一个数据加一小段数据注释了。

数据结构的计数器

如果一个数据结构，在创建和销毁它的单线执行环境之外可见，那么必须要有一个引用计数器，它允许多个用户并行访问这个数据结构，而不当心因为暂时不被使用就消失了。 注意上锁的区别，上锁是为了保持数据结构的一致性，而计数器是一个内存管理技巧，很多数据结构有2级计数器，子类计数器统计子类用户数量，每当子类计数器减至零时，全局计数器减一。

宏，枚举和RTL

  • 宏定义和枚举标签一般大写
  • 在定义相关的常量时，最好用枚举
  • 像函数的宏可以使用小写字母，一般的能写成内联函数的就不要写成像函数的宏。

  • 含有多个语句的宏应该被包含在一个 do-while 代码块里：

    #define macrofun(a, b, c) \
do { \
if (a == 5) \
do_this(b, c); \
} while (0)
  • 使用宏应该避免一下情形：
    • 影响控制流程的宏
    • 依赖于固定名字的本地变量的宏
    • 作为左值的带参数的宏： FOO(x) = y；
    • 忘记了优先级: （记得使用小括号）
    • RTL – register transfer language

内联函数

  • inline 的过度使用会使内核变大，

函数返回值和命令

函数可以返回很多值，比如表面函数执行成功或是失败（-Exxx=失败，0=成功）或者（0=失败，非0=成功）。如果混用这两种表达式是难于发现bug的来源。

  • 请遵循：如果函数是一个动作或者强制性的命令，那么函数返回错误代码整数，
  • 如果是一个判断，那么函数应该返回一个“成功”的布尔值。

不要重复定义宏

计算一个数组的长度
#define ARRAY_SIZE(x) (sizeof(x) / sizeof((x)[0]))
计算某结构体成员的大小
#define FIELD_SIZEOF(t, f) (sizeof(((t*)0)->f))
min()和 max()宏
